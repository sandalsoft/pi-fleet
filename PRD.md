Product Requirements Specification for the Multi-Agent Terminal Orchestration Harness and Real-Time DashboardThe emergence of autonomous agentic systems has necessitated a transition from simple chat-based interfaces to sophisticated orchestration environments that provide deep observability into the cognitive and operational state of artificial intelligence entities. The system defined in this specification, hereafter referred to as the Orchestration Harness, is designed as a modular extension for the pi-agent framework. It provides a generalized, configuration-driven environment where any ad-hoc agent team can be deployed, monitored, and steered in real-time. This architecture leverages the pi-agent core as a minimal, high-performance runtime, extending it with a robust terminal user interface (TUI) and a resource management layer that tracks financial expenditure and temporal progress with granular precision.The fundamental design philosophy of the Orchestration Harness is rooted in transparency and extensibility. By utilizing a "primitives over features" approach, the harness provides the necessary infrastructure for multi-agent collaboration without dictating specific vertical behaviors. Instead, agent personalities, model preferences, and domain expertise are defined in external configuration files and markdown-based system prompts, allowing for rapid reconfiguration of teams to suit any professional workflow.System Architecture and Core ComponentsThe Orchestration Harness operates as a sophisticated layer atop the pi-agent coding harness, recognized for its minimal system prompt and high-efficiency tool execution environment. The architecture is divided into the Gateway for message routing, the Agent Runtime for lifecycle management, and the Surface for rendering the live terminal dashboard. This decoupling ensures that orchestration logic remains independent of specific models, facilitating heterogeneous teams where models like Claude 3.5 Opus can handle reasoning while faster models like Sonnet or Haiku manage specialized tasks.The Extension-Based Orchestration ModelThe system utilizes the pi-agent extension system to register a "Dispatcher" agent that serves as the primary point of contact for the user. This agent interprets a "Brief Markdown File" to identify required skills and specialists. The dispatch logic is implemented as a TypeScript module that hooks into the session lifecycle to spawn and consolidate outputs from multiple specialist agents.ComponentResponsibilityImplementation MechanismHarness CoreTool execution (read, write, bash), session persistence, and provider wiring.pi-agent SDK / pi-mono packages Dispatcher ExtensionInterprets the brief, selects agents, and manages parallel task delegation.agent-team.ts / dispatch_agent tool Dashboard SurfaceReal-time rendering of the agent grid, status indicators, and cost metrics.Textual (Python) or Bubble Tea (Go) TUI framework Resource MonitorTracks token usage, maps to pricing tables, and enforces hard budget/time caps.OpenTelemetry metrics and middleware hooksSafety EngineIntercepts dangerous bash commands and enforces path-based access controls.damage-control.ts regex-based auditing The core multi-agent logic resides in the dispatch_agent tool. When the primary agent determines a task requires a specialist, it invokes this tool with the specialist's name, a task-specific prompt, and relevant context. The harness then initializes the specialist using parameters from the configuration file, creating a branched session that operates without cluttering the main conversation thread.Data Flow and ParallelismThe Orchestration Harness supports parallel processing, allowing multiple specialists to respond simultaneously rather than sequentially. The harness manages concurrent provider requests, streaming responses from each agent into the terminal dashboard in real-time. This is achieved through an asynchronous event-driven model where message_update events from different agent IDs are multiplexed by the TUI layer.To maintain coherence during parallel execution, agents periodically update their "Mental Model" via agent-specific scratchpad files. These files serve as persistent memory, allowing specialists to track patterns and insights across project phases. The TUI reflects these updates live, showing when an agent is "updating its mental model" or "consolidating findings".Ad-Hoc Agent Configuration and DefinitionThe Orchestration Harness allows for ad-hoc deployment of agent teams via external configuration, eliminating the need for hard-coded logic and allowing the system to adapt to any domain.The Team Configuration SchemaThe collective behavior of an agent team is defined in a teams.yaml or settings.json file. Users can set thresholds where the system will automatically begin wrapping up.Soft Constraints: As limits are approached (e.g., 80% of budget), the harness injects warnings into system prompts, directing agents to prioritize consolidation over exploration.Hard Constraints: Upon hitting a hard limit, the harness terminates tool executions (after saving work) and triggers a final summarization turn to produce artifacts like project memos or decision maps.Skills, Tools, and Expertise InjectionThe harness expands capabilities through a structured way for agents to acquire new tools and domain knowledge on demand.The Skill SystemA "Skill" is a modular package consisting of a markdown instruction file (SKILL.md) and optional TypeScript-based tools. These can be referenced dynamically in configuration or injected via front matter skill keys to bypass the need for agents to "decide" to load a tool, saving tokens and latency.Agents maintain "persistent scratchpad files" to track domain-specific notes across turns. The harness also supports "Project Context Files" (AGENTS.md) loaded at startup to provide global instructions applicable to all agents in the workspace.Task Flow and Orchestration ParadigmsThe harness provides two primary methods for managing work flow between agents:The Dispatcher ModelThe primary agent acts as a manager, creating task briefs and delegating work via the dispatch_agent tool. This is ideal for ad-hoc tasks where the sequence of operations is not known in advance.The Sequential Chain ModelThe agent-chain extension allows for deterministic pipelines where the output of one agent ($INPUT) is passed to the next. Workflows are defined in an agent-chain.yaml file and tracked via the TUI dashboard.Safety, Guardrails, and Damage ControlThe harness includes a "Damage Control" engine that monitors agent activity in real-time to ensure safe interaction with the local system.Command Auditing: Intercepts shell-based tool calls. Rules in damage-control-rules.yaml can block destructive patterns (e.g., rm -rf /) or force user confirmation for network-accessing commands.Path Isolation: Supports "Workspace Isolation," forcing agents to work in separate branches or temporary directories to prevent contamination of the main project worktree.
